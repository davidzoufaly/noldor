# Graph Integration

Noldor's SDD detectors (Detector 13 onward) consume a knowledge graph
generated by [`graphify`](https://pypi.org/project/graphifyy/). This page
describes the substrate: which graph file to read for what, the
staleness gate, and the path → FD lookup pattern detectors use.

The graph itself lives in `graphify-out/`, regenerated via the
`/graphify` skill (toon files via `pnpm toon`). It contains hyperedges,
god nodes, community structure, and cross-package bridges that no file
scanning replicates.

## Which file to use

| File                            | When to use                                                                                                                                         |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| `graph.brainstorm-summary.toon` | **Default starting point.** Quick orientation, brainstorming, "what packages exist", architecture questions. Read this first in every conversation. |
| `graph.brainstorm.toon`         | **Deep work.** Refactoring, tracing cross-cutting concerns, community-level design, finding god nodes. Use when summary isn't enough.               |
| `GRAPH_REPORT.md`               | **Audit.** God node analysis, surprising connections, community cohesion scores, knowledge gaps.                                                    |
| `graph.json`                    | **Programmatic queries only.** Don't read raw — use toon files instead. Only for scripts or when you need exact IDs/edge weights.                   |
| `graph.html`                    | **Visual exploration.** Open in browser, click communities. Not for LLM consumption.                                                                |

## Rules — when to read what

- **Before any codebase exploration**, read
  `graph.brainstorm-summary.toon`. This includes subagent prompts —
  always instruct them to read the toon first.
- Escalate to `graph.brainstorm.toon` only when the summary doesn't
  answer your question.
- Never read `graph.json` directly unless doing programmatic work — the
  toon files exist specifically to save tokens.
- Outside the pre-release sweep, regenerate the graph on demand (e.g.
  before deep refactor work, or when brainstorming needs accurate
  structure). Skip per-feature regen — bounded drift between releases is
  acceptable.
- After running `/graphify`, always run `pnpm toon` to regenerate toon
  files.

## Pre-release sweep

The pre-release sweep — `/graphify` → `/refactor` against the new
`GRAPH_REPORT.md` → verify README → `/graphify` again — is encoded in
the [`/release-sweep`](../../.claude/skills/release-sweep/SKILL.md) skill and
required before any minor or major `pnpm release`. See
[`versioning.md`](versioning.md) "Pre-release sweep" for the release-flow
contract.

## graph-fd-lookup substrate

Detectors that need to map a code file to its owning FD use the path → FD
lookup substrate at
[`../../src/garden/graph-fd-lookup.ts`](../../src/garden/graph-fd-lookup.ts).
Three exports cover the pattern:

- **`loadFreshGraphOrWarn(graphPath, srcRoots)`** — load `graph.json`
  with the staleness gate baked in. Returns `{ ok: true, graph }` or
  `{ ok: false, gap }` so the consuming detector can propagate the
  meta-gap as its only finding when the graph is missing or stale.
- **`buildFileToFdsMap(features)`** — invert every FD's `links.code`
  array into a `Map<path, Set<slug>>`. Co-ownership is real (a meta-FD
  can name a file already owned by a primary FD), so the value is a set,
  not a single slug.
- **`getFdOwnersForFile(filePath, map)`** — look up owners for a file,
  walking ancestor directories so a `links.code` entry like
  `packages/sample-scenes` covers
  `packages/sample-scenes/src/empty-room.ts`. Owners from a direct match
  and ancestor-directory matches are unioned.

Used by:

- Detector 13 (co-tag suggestion).
- Detector 9 (orphan owner suggestion) when it ships — see
  [`../backlog.md`](../backlog.md) "SDD Detector 9".
- Detector 10 (untagged-test slug suggestion) when it ships — see
  [`../backlog.md`](../backlog.md) "SDD Detector 10".

## Staleness gate — graph older than latest source commit

Detectors that consume `graph.json` check its mtime against the largest
mtime of files under tracked source roots (typically
`['packages', 'apps', 'scripts']`). If any source file is newer than the
graph, the detector skips with a single meta-gap that includes regen
instructions, rather than emitting stale findings.

Implemented in `loadFreshGraphOrWarn` (see above). Mtime-based staleness
is intentionally cheap; the pre-release sweep already forces a fresh
`/graphify` regen, so any false-stale (e.g. after `git checkout`) is
harmless — it forces a regen, which is the right outcome anyway.

## Dependency on graphify v0.4.20+

Path normalization (cross-package edge accuracy, no double-counted edges)
requires graphify ≥ 0.4.20. The bug was fixed upstream in commit
`a507c97` (`fix #414 JS imports_from path normalisation`); lower versions
emit edges keyed by inconsistent paths and silently break the substrate.
Pin the version in adopting repos (use ≥ v0.7.8, the release that includes this fix).
