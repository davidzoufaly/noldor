# Documentation Conventions

This page collects the framework's documentation rules: feature-doc / test-tag conventions, transclusion markers, the generated-do-not-edit sentinel, the auto-regenerated how-to index pipeline, and the Diataxis taxonomy that organizes user-facing docs.

## Commands

| Trigger                         | What it does                                                                                    |
| ------------------------------- | ----------------------------------------------------------------------------------------------- |
| `pnpm noldor sync doc-links`    | Crawl `<!-- @feature: <slug> -->` tags, populate FD `links.docs`. Pre-commit hook.              |
| `pnpm noldor sync test-links`   | Crawl `// @tests: <slug>` tags, populate FD `links.tests`. Pre-commit hook.                     |
| `pnpm noldor validate features` | Reject unknown `@feature:` / `@tests:` slugs. Pre-commit hook.                                  |
| `pnpm noldor docs transclude`   | Sync `<!-- example:start <name> --> ... <!-- example:end -->` blocks from `packages/examples/`. |
| `pnpm noldor docs howto`        | Regenerate `docs/user/how-to/index.md`. Don't hand-edit the index.                              |
| `pnpm noldor docs api`          | Regenerate `docs/user/reference/api/**` (typedoc). Don't hand-edit.                             |
| `pnpm noldor docs check`        | Verify no broken transclusion, no orphan `@feature:` / `@tests:` tags, no dangling links.       |
| `pnpm docs:build`               | Composite: `docs:api && docs:howto && docs:transclude && sync:doc-links && docs:check && fmt`.  |

## Sentinels (do not edit by hand)

| Marker                                                  | Owned by                                                                    |
| ------------------------------------------------------- | --------------------------------------------------------------------------- |
| `<!-- generated: do-not-edit -->`                       | `pnpm noldor docs api`, `pnpm noldor docs howto` — clobbered on next build. |
| `<!-- example:start ... -->` ... `<!-- example:end -->` | `pnpm noldor docs transclude` — body is overwritten.                        |
| `<!-- generated: resources -->`                         | `pnpm noldor sync fd-resources` — FD body Resources block.                  |
| `<!-- @prs-since-last-release: <slug> -->`              | `pnpm dashboard` render path — expanded to PR bullet list at view time.     |

## `@feature:` tag — every doc tagged

- **Every new feature should appear in at least one tutorial or explanation when relevant.** Tag the doc with `<!-- @feature: <slug>[, <slug>] -->` near the top. `pnpm noldor sync doc-links` populates feature MD `links.docs` from these tags. `pnpm noldor validate features` rejects unknown slugs (mirror of `@tests:` rule).

## `@tests:` tag — every test file tagged

- **Every test file must carry `// @tests: <feature-slug>[, <feature-slug>]` as its first non-import line.** `pnpm noldor sync test-links` populates feature MD `links.tests` from these tags. `pnpm noldor validate features` rejects unknown slugs. Both run in the pre-commit hook.

## Tutorial code blocks transclude from a tested examples source

- **Tutorial code blocks transclude from a unit-tested examples location** (wherever the consumer keeps runnable examples) — never inline. Use `<!-- example:start <name> -->` ... `<!-- example:end -->` markers; `pnpm noldor docs transclude` syncs the body. Editing inside the marker body is wasted work — it gets overwritten on next `pnpm docs:build`.

## Generated docs — do not hand-edit

- **Generated docs carry `<!-- generated: do-not-edit -->` and must not be hand-edited.** This includes `docs/user/reference/api/**` (typedoc) and `docs/user/how-to/index.md`. Hand-edits are clobbered on next build.

## How-to index — auto-regenerated

- `docs/user/how-to/index.md` is auto-regenerated by `pnpm noldor docs howto` from feature MD frontmatter. Don't hand-edit it. The README likewise carries no auto-generated feature listing — the canonical feature index lives at `docs/user/how-to/index.md`.

## Diataxis structure

- User-facing documentation follows the [Diataxis](https://diataxis.fr/) framework: **tutorials** (learning-oriented), **how-to guides** (task-oriented), **reference** (information-oriented), **explanation** (understanding-oriented). Tag each doc accordingly and place it under `docs/user/<type>/`. See the `diataxis` skill for classification rules when a piece of content is ambiguous. The four-doc-type taxonomy keeps user-facing docs predictable for both humans and agents.
